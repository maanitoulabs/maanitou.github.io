1 Introduction

Linux device drivers are a type of kernel module that allow the kernel to access devices connected to the system. In the following we will demonstrate how to write, cross-compile and test loadable kernel modules for the RPi.

2 Building the Linux kernel

The Linux distribution installed on the RPi is a Poky distribution with kernel version number 5.4.83:

root@raspberrypi0-wifi:~# uname -r
5.4.83

Firstly, the sources of the Poky distribution installed on the RPi should be copied to the host machine in, say, ~/src/rpi-5.4.83/.

cd ~/src/rpi-5.4.83

Then, we need to configure the correct target version using a kernel configuration file. The configuration file that we will use is the default configuration file for the RPi, rpizw_defconfig, and it is found in the ./arch/arm/configs/ folder. The configuration file can be examined and modified via a UI that can be launched with make menuconfig.

$ make ARCH=arm CROSS_COMPILE=arm-poky-linux-gnueabi- rpizw_defconfig

Since we are building modules for a target architecture and target OS which are different from the host’s we need to 1) specify the target ARM architecture with the ARCH=arm argument, 2) use the CROSS_COMPILE=... argument to specify which gcc-prefix to use for cross-compilation. Here, we use the compiler toolchain provided by the Poky distribution.

We prepare the Linux kernel source tree for building kernel modules by running the following command:

make ARCH=arm CROSS_COMPILE=arm-poky-linux-gnueabi- modules_prepare

Optionally, we may want to build the kernel image (zImage) and kernel modules. However, it is not needed for cross-compilation of kernel modules, only if we want to install a modified kernel on the RPi.

make -j 8 ARCH=arm CROSS_COMPILE=arm-poky-linux-gnueabi- zImage modules

The generated Linux kernel is a single binary located in the boot folder:

$ ls -sh ./arch/arm/boot/zImage
4.2M ./arch/arm/boot/zImage

Finally, we may want to clean up everything by running make distclean and make clean.

Now that we have a precompiled kernel source tree, we can proceed to writing a kernel module.

3 Creating a kernel module

Let us write a Hello World kernel module. The code and makefile are located in the code/hello_module/ directory.

#include <linux/module.h>

static int __init loader(void)
{
    pr_info("The Hello World module has been inserted.\n");
    return 0;
}

static void __exit unloader(void)
{
    pr_info("The Hello World module has been removed.\n");
}

module_init(loader);
module_exit(unloader);

MODULE_DESCRIPTION("HelloWorld module");
MODULE_AUTHOR("maanitou");
MODULE_LICENSE("GPL");

4 Building the kernel module

The source code of our module is outside the kernel source tree, so the method for building our module is called out-of-tree building. This method will produce a loadable kernel module.

A minimal makefile for building the hello module is shown below.

obj-m := hello.o

modules clean:
    make \
    ARCH=arm \
    CROSS_COMPILE=arm-poky-linux-gnueabi- \
    -C ~/src/rpi-5.4.83 \
    M=$(shell pwd) $@
  • ARCH=arm and CROSS_COMPILE=arm-poky-linux-gnueabi are the target architecture and cross-compiler gcc-prefix,
  • -C ~/src/rpi-5.4.83 specifies the directory of the linux kernel tree that was built in the previous section,
  • M=$(shell pwd) is used to specify the directory in which we are building our external module (the directory where the module sources and the makefile are located).
  • We have two targets, modules and clean, meaning that we can either build the external module running make modules (or just make since the modules target is the default), or remove all generated files in the module directory with make clean.
  • obj-m := hello.o means that we want to build the object file hello.o from hello.c. The -m of obj-m means that we want to link the object file into a kernel loadable module hello.ko.

A more thorough explanation of how to write makefiles for external modules is given in the Building External Modules section of the Linux Documentation, or in [LDDD, Chapter 2].

5 Testing the kernel module

The following script transfers our module hello.ko to the RPi, into a dedicated folder, here 03_kernel_modules.

#!/bin/sh

REMOTE_HOST=root@10.9.8.2
REMOTE_INSTALL_DIR=03_kernel_modules

set -x 

ssh $REMOTE_HOST "mkdir -p $REMOTE_INSTALL_DIR"
scp hello.ko $REMOTE_HOST:$REMOTE_INSTALL_DIR

Then, after ssh-ing to the RPi, we can insert and remove the new kernel module, and check, using dmesg, that everything worked properly.

$ insmod hello.ko
$ rmmod hello.ko
$ dmesg | tail -n 2
[14206.879122] The Hello World module has been inserted.
[14210.204319] The Hello World module has been removed.
